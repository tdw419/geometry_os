# Milestone v1.3: Multi-Boot

**Status:** ✅ SHIPPED 2026-03-09
**Phases:** 9-11
**Total Plans:** 12

## Overview

Boot multiple PixelRTS containers simultaneously with network connectivity. Users can boot multiple containers with a single CLI command, designate primary/helper relationships for ordered boot, and enable inter-container networking without root privileges.

## Phases

### Phase 9: Core Multi-Boot Infrastructure

**Goal**: Users can boot multiple containers simultaneously with automatic resource allocation
**Depends on**: Phase 8.1 (existing boot infrastructure)
**Requirements**: MULTI-01, MULTI-02, MULTI-03, STATUS-01, STATUS-02
**Plans**: 5 plans

Plans:
- [x] 09-01: ResourceAllocator for VNC ports and socket paths
- [x] 09-02: MultiBootManager with concurrent boot orchestration
- [x] 09-03: Cleanup on partial failure (compensating transactions)
- [x] 09-04: `pixelrts ps` status command
- [x] 09-05: CLI multi-file boot support

**Success Criteria:**
1. User can run `pixelrts boot a.rts.png b.rts.png` and both containers start
2. Each container gets unique VNC port and socket path (no conflicts)
3. If one container fails to boot, already-started containers are cleaned up
4. User can run `pixelrts ps` to see running containers with name, VNC port, state
5. Status shows container states (launching/running/stopped)

### Phase 10: Boot Ordering & Dependencies

**Goal**: Users can designate primary/helper containers with ordered startup and shutdown
**Depends on**: Phase 9 (multi-boot foundation)
**Requirements**: ORDER-01, ORDER-02, ORDER-03
**Plans**: 4 plans

Plans:
- [x] 10-01: Container role designation (primary/helper enum, --primary CLI flag)
- [x] 10-02: Boot dependency ordering with wait logic (primary first, helpers wait)
- [x] 10-03: Reverse-order graceful shutdown (stop_all_ordered)
- [x] 10-04: Boot progress visibility (progress_callback, CLI output)

**Success Criteria:**
1. User can designate one container as primary (starts first)
2. Helper containers wait until primary is running before starting
3. Shutdown reverses boot order (helpers stop first, primary last)
4. User can observe boot order progress in CLI output

### Phase 11: Virtual Networking

**Goal**: Containers can communicate with each other without root privileges
**Depends on**: Phase 10 (ordered multi-boot)
**Requirements**: NET-01, NET-02
**Plans**: 3 plans

Plans:
- [x] 11-01: VirtualNetwork class with QEMU socket netdev (multicast mesh)
- [x] 11-02: NetworkMode enum extension (SOCKET_MCAST, SOCKET_STREAM)
- [x] 11-03: Graceful degradation on network failure with fallback to USER mode

**Success Criteria:**
1. Containers can ping/connect to each other over virtual network
2. Networking works without root or CAP_NET_ADMIN privileges
3. Network setup failure falls back gracefully to isolated mode

## Milestone Summary

**Key Decisions:**
- **09-01:** VNC port range 5900-5999 (100 ports), thread-safe allocation, UUID5 deterministic IDs
- **09-02:** asyncio.gather for concurrent boot, run_in_executor for sync-to-async bridge, ContainerState enum for lifecycle
- **09-03:** Compensating transaction cleanup on partial failure, cleanup_on_failure=True by default
- **09-04:** State file at /tmp/pixelrts/containers.json, table output with NAME/STATE/VNC/PID columns
- **09-05:** CLI multi-file boot via nargs='+', exit code 0 on partial success
- **10-01:** ContainerRole enum (PRIMARY/HELPER) for ordered boot, default HELPER role, --primary CLI flag
- **10-02:** Ordered boot pattern (primary first, wait for RUNNING, helpers concurrently)
- **10-03:** Ordered shutdown pattern (helpers first, primary last), stop_all_ordered() method
- **10-04:** Progress callback pattern for ordered boot visibility (primary_start, primary_ready, helpers_start, helper_ready)
- **11-01:** VirtualNetwork class for QEMU socket netdev with multicast, default mcast 230.0.0.1:1234
- **11-02:** NetworkMode enum extension (SOCKET_MCAST, SOCKET_STREAM), QemuConfig.socket_config field
- **11-03:** Graceful network fallback to USER mode on failure, network_fallback flag in ContainerInfo, --network CLI option

**Issues Resolved:**
- None (clean implementation)

**Technical Debt:**
- SOCKET_STREAM mode raises NotImplementedError (future work)
- Network fallback warning could be more detailed
