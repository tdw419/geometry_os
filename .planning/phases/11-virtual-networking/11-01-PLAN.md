  <files>systems/pixel_compiler/boot/virtual_network.py</files>
  <action>
Create a new file `systems/pixel_compiler/boot/virtual_network.py` with:

1. **VirtualNetworkConfig dataclass:**
   - `mcast_addr: str = "230.0.0.1"` - Multicast group address
   - `mcast_port: int = 1234` - UDP port for multicast
   - `enabled: bool = True` - Whether virtual networking is enabled

2. **NetworkSetupError exception:**
   - Custom exception for network setup failures

3. **VirtualNetwork class:**
   - `__init__(self, config: Optional[VirtualNetworkConfig] = None)`
   - `build_netdev_args(self, device_id: str = "net0") -> List[str]`
     - Returns QEMU arguments like: `["-netdev", "socket,id=net0,mcast=230.0.0.1:1234", "-device", "virtio-net-pci,netdev=net0"]`
   - `is_available(self) -> bool` - Always returns True for socket networking (no privilege check needed)
   - `get_network_info(self) -> Dict[str, Any]` - Returns config as dict for status display

Key implementation details:
- Use QEMU socket netdev with multicast: `-netdev socket,id=ID,mcast=ADDR:PORT`
- Pair with virtio-net device: `-device virtio-net-pci,netdev=ID`
- No root privileges required (this is the key advantage over TAP/bridge)
- Default multicast address 230.0.0.1:1234 allows all containers on same host to communicate

DO NOT use TAP or bridge networking - those require root/CAP_NET_ADMIN.
DO NOT create actual network interfaces - just generate QEMU arguments.

    - `python -c "from systems.pixel_compiler.boot.virtual_network import VirtualNetwork; vn = VirtualNetwork(); assert vn.is_available() == True"`
