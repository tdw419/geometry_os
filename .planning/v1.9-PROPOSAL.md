# v1.9 Milestone Proposal: Distributed Boot

**Proposed:** 2026-03-10
**Depends on:** v1.8 Network Boot Substrate (complete)

## Goal

Boot containers on remote peers and migrate running containers across the mesh.

## Why This Milestone

v1.8 established peer discovery and state sync. The natural next step is to actually **use** the mesh - boot containers on remote nodes, migrate workloads, and balance load across the cluster.

## Proposed Features

### Phase 25: Remote Boot Protocol

**Goal:** Boot a container on a remote peer via mesh

**Requirements:**
- **DIST-01**: Boot container on remote peer
- **DIST-02**: Remote boot status tracking
- **DIST-03**: Remote boot cancellation

**Success Criteria:**
1. `pixelrts boot --peer <hostname> file.rts.png` boots on remote node
2. Boot progress streams back to caller
3. VNC port forwarded to caller for console access
4. Remote boot failures propagate to caller

**Estimated Plans:** 4

---

### Phase 26: Container Migration

**Goal:** Migrate running container to another peer

**Requirements:**
- **MIGR-01**: Checkpoint container state
- **MIGR-02**: Transfer state to target peer
- **MIGR-03**: Restore container on target
- **MIGR-04**: Network reconnection after migration

**Success Criteria:**
1. `pixelrts migrate <container> <target-peer>` initiates migration
2. VM memory state transferred via mesh
3. Container resumes on target with same state
4. VNC connection re-established automatically

**Estimated Plans:** 5

---

### Phase 27: Load Balancing

**Goal:** Automatic container placement based on peer capacity

**Requirements:**
- **LB-01**: Peer resource reporting (CPU, memory, disk)
- **LB-02**: Capacity-aware boot routing
- **LB-03**: Migration triggers (overload detection)

**Success Criteria:**
1. `pixelrts boot --auto-place file.rts.png` selects best peer
2. Peer metrics exchanged via mesh
3. Overloaded peers trigger automatic migration
4. Dashboard shows cluster-wide resource usage

**Estimated Plans:** 4

## Alternative Milestone Options

If distributed boot is too ambitious, consider:

### Option B: Mesh Security (v1.9b)

- MESH-01: Peer authentication via Ed25519
- MESH-02: Encrypted peer communication
- MESH-03: Mesh-wide boot orchestration

### Option C: Container Templates (v1.9c)

- Pre-configured container templates
- One-click boot from template
