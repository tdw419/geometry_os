## Learnings

- Strategy pattern ideal for reduction strategies (extensible)
- flock critical for cross-process atomicity (not threading.Lock)
- File-based storage simplest for MVP (no database deps)
- Task lifecycle: PENDING→CLAIMED→IN_PROGRESS→COMPLETED/FAILED
- 4 task types mapped to 4 reduction strategies
- TaskBoard uses per-board lock file for atomic ops
- parent_id stored in task for subtask relationships
- get_strategy() factory returns strategy instance for flexible use
- MajorityVoteStrategy uses vote_key parameter
